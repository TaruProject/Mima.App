### Critical Bug: Infinite Recursion

**`src/hooks/useAudioPlayback.ts:26-29`** — `updateTtsStatus` calls itself recursively instead of `setTtsStatus`:

```typescript
// BROKEN — line 28
const updateTtsStatus = useCallback((status: TtsStatus) => {
  ttsStatusRef.current = status;
  updateTtsStatus(status); // ← INFINITE RECURSION: should be setTtsStatus(status)
}, []);
```

**Impact**: When `isOptimizationEnabled()` is `true` (the default), any call to `playCached()` triggers `updateTtsStatus('loading')` which stack-overflows immediately. This means the entire TTS optimization path is **completely broken in production**.

**Fix**: Change `updateTtsStatus(status)` → `setTtsStatus(status)` on line 28.

### Root Cause of 3-Second Delay

1. **Bug above** prevents the cache path from ever executing → all playback falls through to `play()` legacy path
2. **Legacy `play()` path** (line 239-305): creates audio element → unlocks iOS → fetches from server → waits for full response → decodes → plays = **3-5 seconds**
3. **No preloading in legacy path** — the `preload()` call in Chat.tsx only runs when `isOptimizationEnabled()` returns true, but even when it does, `playCached` crashes, so the preload was wasted

### Phase 5: Server-Side TTS Cache Headers (5 min)

#### File: `server.ts`

**Current** (line 1999):

```typescript
res.setHeader('Cache-Control', 'private, max-age=86400, stale-while-revalidate=3600');
```

This is already correct for browser HTTP cache. However, the frontend uses `cache: 'no-store'` in `fetchAudio()` (line 147), which **bypasses the HTTP cache entirely**.

**Two options**:

**Option A (Recommended)**: Keep `cache: 'no-store'` and rely on IndexedDB cache. The HTTP cache is redundant when IndexedDB works. No server change needed.

**Option B**: Change `cache: 'no-store'` to `cache: 'default'` when optimization is enabled, allowing browser HTTP cache as an additional layer. This adds complexity and the IndexedDB cache already covers this.

**Decision**: Option A — no server changes required. The IndexedDB cache + preloading eliminates the need for HTTP cache. The existing `Cache-Control` header is a nice-to-have for debugging but doesn't affect the optimization path.

### Phase 6: Preload Optimization (10 min)

#### File: `src/pages/Chat.tsx`

**Current preload logic** (lines 760-793):

- Runs on every `messages` or `voiceId` change
- Skips messages already in `ready` or `loading` status
- Uses `requestIdleCallback` with 3s timeout
- Preloads ALL assistant messages (no limit)

**Issues**:

1. No deduplication of in-flight preloads — multiple `preload()` calls for the same message can race
2. No limit on concurrent preloads — 10 messages = 10 simultaneous ElevenLabs API calls (rate limit risk)
3. Welcome message is correctly excluded (`!msg.isWelcome`)

**Improvements**:

1. Add a preload tracking `Set` to avoid duplicate in-flight requests:

```tsx
const preloadInFlightRef = useRef<Set<string>>(new Set());
```

2. Limit concurrent preloads to 2:

```tsx
const MAX_CONCURRENT_PRELOADS = 2;

useEffect(() => {
  if (!isOptimizationEnabled()) return;

  const assistantMessages = messages.filter(
    (msg) => msg.role === 'assistant' && !msg.isWelcome && msg.text.trim()
  );
  if (assistantMessages.length === 0) return;

  const schedulePreload = () => {
    let activeCount = preloadInFlightRef.current.size;

    for (const msg of assistantMessages) {
      if (activeCount >= MAX_CONCURRENT_PRELOADS) break;

      const key = msg.id.toString();
      if (ttsCacheStatuses[key] === 'ready' || ttsCacheStatuses[key] === 'loading') continue;
      if (preloadInFlightRef.current.has(key)) continue;

      preloadInFlightRef.current.add(key);
      activeCount++;

      preload(msg.text, voiceId).finally(() => {
        preloadInFlightRef.current.delete(key);
        checkCacheStatus(msg.text, voiceId).then((status) => {
          setTtsCacheStatuses((prev) => {
            if (prev[key] === status) return prev;
            return { ...prev, [key]: status };
          });
        });
      });
    }
  };

  if ('requestIdleCallback' in window) {
    (window as any).requestIdleCallback(schedulePreload, { timeout: 3000 });
  } else {
    setTimeout(schedulePreload, 100);
  }
}, [messages, voiceId]);
```

Note: `.then()` changed to `.finally()` so the tracking set is cleaned up even on failure.

1. **Immediate**: Set `localStorage.mima_tts_optimization = 'legacy'` (or remove key) → falls back to original `play()` path
2. **Code revert**: All changes are backward-compatible; reverting any file restores previous behavior
3. **Data**: IndexedDB `mima_tts_cache` can be safely deleted by user or programmatically — no data loss

## 8. Future Enhancements (Out of Scope)

- **Server-side TTS cache** (Redis/in-memory): Cache audio blobs server-side keyed by `SHA256(text+voiceId)` to avoid duplicate ElevenLabs calls across users
- **Streaming audio**: Use ElevenLabs streaming API to start playback before full generation completes (could reduce perceived latency to <500ms for cache misses)
- **Adaptive model**: Use `eleven_monolingual_v1` for English-only text (faster) and `eleven_multilingual_v2` only when needed
- **Cache warming**: Pre-generate TTS for system/welcome messages on app load
- **Audio compression**: Store Opus/WebM instead of MP3 in IndexedDB for smaller cache footprint
