## Phase 1: Startup Validation (Foundation)

### What

Replace the deferred `setTimeout()` env validation with **synchronous, blocking startup validation**. Server exits immediately if `SESSION_SECRET` is missing or <32 chars.

### Implementation

**`server.ts` lines 64-109** — Replace the deferred validation block:

```
// BEFORE (deferred, non-blocking):
setTimeout(() => { ... }, delay);

// AFTER (blocking, fail-fast):
const SESSION_SECRET = process.env.SESSION_SECRET;
if (!SESSION_SECRET || SESSION_SECRET.length < 32) {
  console.error('[FATAL] SESSION_SECRET missing or <32 chars. Server cannot start.');
  process.exit(1);
}
```

- Remove both fallbacks: `'mima-default-fallback-secret-32-chars-long'` (line 141) and `'mima-session-fallback-secret'` (line 200)
- `SESSION_SECRET` must be validated BEFORE `ENCRYPTION_KEY` derivation and BEFORE `session()` middleware setup
- Keep the deferred validation for OTHER env vars (GEMINI_API_KEY, etc.) — they log warnings but don't block startup
- Add `mima_security_v2` check: if explicitly set to `"false"`, skip hardened validation with `[SECURITY] Legacy mode — not hardened` warning

### New env var

```
SUPABASE_DB_URL=postgresql://postgres:[password]@db.[ref].supabase.co:5432/postgres
```

Add to `requiredEnvVars` and `.env.example`.

## Phase 2: Session Store Migration (MemoryStore → PostgreSQL)

### What

Replace implicit `MemoryStore` with `connect-pg-simple` backed by Supabase's direct PostgreSQL connection.

### Dependencies to install

```bash
npm install connect-pg-simple pg
npm install -D @types/connect-pg-simple @types/pg
```

### Database migration

**New file:** `supabase/migrations/20260422000000_create_sessions_table.sql`

```sql
CREATE TABLE IF NOT EXISTS public.session (
  sid VARCHAR NOT NULL COLLATE "default",
  sess JSON NOT NULL,
  expire TIMESTAMP(6) NOT NULL
);

ALTER TABLE public.session ADD CONSTRAINT session_pkey PRIMARY KEY (sid) NOT DEFERRABLE INITIALLY IMMEDIATE;

CREATE INDEX IF NOT EXISTS IDX_session_expire ON public.session (expire);

-- Prune expired sessions automatically (runs every hour)
CREATE OR REPLACE FUNCTION public.cleanup_expired_sessions()
RETURNS void AS $$
BEGIN
  DELETE FROM public.session WHERE expire < NOW();
END;
$$ LANGUAGE plpgsql;

-- Use pg_cron if available, otherwise rely on connect-pg-simple's pruneInterval
```

### Implementation in `server.ts`

Replace lines 198-211 (session middleware):

```typescript
import pg from 'pg';
import connectPgSimple from 'connect-pg-simple';

const pgPool = new pg.Pool({
  connectionString: process.env.SUPABASE_DB_URL,
  // Supabase requires SSL for direct connections
  ssl: { rejectUnauthorized: false },
  // Connection pooling
  max: 10,
  idleTimeoutMillis: 30000,
  connectionTimeoutMillis: 5000,
});

const PgSessionStore = connectPgSimple(session);

app.use(
  session({
    store: new PgSessionStore({
      pool: pgPool,
      tableName: 'session',
      schemaName: 'public',
      // Prune expired sessions every 15 minutes
      pruneSessionInterval: 900,
      // TTL 7 days
      ttl: 7 * 24 * 60 * 60,
    }),
    secret: SESSION_SECRET, // No fallback — validated at startup
    resave: false,
    saveUninitialized: false,
    name: 'mima.session',
    cookie: {
      secure: IS_PROD,
      sameSite: IS_PROD ? 'none' : 'lax',
      maxAge: 7 * 24 * 60 * 60 * 1000, // 7 days (match TTL)
      httpOnly: true,
    },
  })
);
```

### Rollback

When `mima_security_v2=false`:

- Skip `connect-pg-simple` setup
- Use `MemoryStore` (default) with 24h cookie maxAge
- Log: `[SECURITY] Legacy mode — using MemoryStore`

### Session data compatibility

The `session` table stores the full session JSON blob. Existing in-memory sessions will be lost on first deploy (acceptable — sessions are transient, users re-auth via Supabase JWT).

## Phase 3: AES-CBC → AES-GCM Encryption Migration

### What

Replace AES-256-CBC with AES-256-GCM. Store IV + Auth Tag alongside ciphertext for authenticated encryption.

### Key derivation changes

**Before (static salt):**

```typescript
ENCRYPTION_KEY = crypto.scryptSync(secret, 'salt', 32);
```

**After (per-deployment salt):**

```typescript
// Derive a deployment-specific salt from SESSION_SECRET itself
const DEPLOYMENT_SALT = crypto.scryptSync(SESSION_SECRET, 'mima-encryption-salt-v2', 16);
ENCRYPTION_KEY = crypto.scryptSync(SESSION_SECRET, DEPLOYMENT_SALT, 32);
```

This makes the salt deterministic per-deployment (so encrypt/decrypt work across restarts) but unique per `SESSION_SECRET` value — not the global `'salt'` string.

### New encrypt/decrypt functions

**Format:** `iv:hex:authTag:hex:ciphertext:hex` (was: `iv:hex:ciphertext:hex`)

```typescript
function encrypt(plaintext: string): string {
  const iv = crypto.randomBytes(12); // GCM standard IV length
  const cipher = crypto.createCipheriv('aes-256-gcm', ENCRYPTION_KEY, iv);
  let encrypted = cipher.update(plaintext, 'utf8');
  encrypted = Buffer.concat([encrypted, cipher.final()]);
  const authTag = cipher.getAuthTag();
  return iv.toString('hex') + ':' + authTag.toString('hex') + ':' + encrypted.toString('hex');
}

function decrypt(ciphertext: string): string {
  const parts = ciphertext.split(':');

  // Detect format: 2 parts = legacy CBC, 3 parts = GCM
  if (parts.length === 2) {
    return decryptCBC(ciphertext); // Backward compat
  }

  const iv = Buffer.from(parts[0], 'hex');
  const authTag = Buffer.from(parts[1], 'hex');
  const encrypted = Buffer.from(parts[2], 'hex');
  const decipher = crypto.createDecipheriv('aes-256-gcm', ENCRYPTION_KEY, iv);
  decipher.setAuthTag(authTag);
  let decrypted = decipher.update(encrypted);
  decrypted = Buffer.concat([decrypted, decipher.final()]);
  return decrypted.toString('utf8');
}

// Legacy CBC decrypt for tokens encrypted before migration
function decryptCBC(ciphertext: string): string {
  const textParts = ciphertext.split(':');
  const iv = Buffer.from(textParts.shift()!, 'hex');
  const encryptedText = Buffer.from(textParts.join(':'), 'hex');
  // Derive legacy key with old static salt for backward compat
  const legacyKey = crypto.scryptSync(SESSION_SECRET, 'salt', 32);
  const decipher = crypto.createDecipheriv('aes-256-cbc', legacyKey, encryptedText);
  let decrypted = decipher.update(iv); // Wait — CBC format is iv:ciphertext
  // ... proper legacy CBC decrypt
  decrypted = Buffer.concat([decrypted, decipher.final()]);
  return decrypted.toString('utf8');
}
```

### Migration strategy (zero-downtime)

1. **New `encrypt()` calls** produce GCM format (`iv:authTag:ciphertext`)
2. **`decrypt()` detects format** by part count: 2 parts = legacy CBC, 3 parts = GCM
3. **Re-encryption happens lazily**: when `persistGoogleTokens()` is called (token refresh, new OAuth), the tokens are re-encrypted with GCM
4. No batch migration needed — all tokens naturally re-encrypt within the access token refresh cycle (1 hour)

### What gets stored

- `user_google_tokens.tokens` column: already TEXT, stores the `iv:authTag:ciphertext` hex string
- No schema change needed for `user_google_tokens` — the IV and auth tag are embedded in the encrypted blob
- This avoids a migration on the Supabase table

### Rollback

When `mima_security_v2=false`:

- Use legacy `encryptCBC()`/`decryptCBC()` functions
- GCM-encrypted tokens won't be decryptable in legacy mode (acceptable — user must re-auth Google)

## Phase 4: Move `src/services/` → `server/services/`

### What

Move the 3 backend-only service files out of `src/` to prevent accidental frontend imports and eliminate `process.env`/`node:crypto` from the Vite-compiled bundle.

### File moves

| From                                     | To                                          |
| ---------------------------------------- | ------------------------------------------- |
| `src/services/userPreferencesService.ts` | `server/services/userPreferencesService.ts` |
| `src/services/userMemoryService.ts`      | `server/services/userMemoryService.ts`      |
| `src/services/userTaskService.ts`        | `server/services/userTaskService.ts`        |

### Changes in moved files

**Replace `process.env` references** with module-level variables set via an `init()` function:

```typescript
// server/services/userMemoryService.ts
import { createHash } from 'node:crypto';
import { createClient, SupabaseClient } from '@supabase/supabase-js';

let supabaseUrl = '';
let supabaseServiceKey = '';

export function initUserService(url: string, key: string) {
  supabaseUrl = url;
  supabaseServiceKey = key;
}

function getSupabase(): SupabaseClient {
  return createClient(supabaseUrl, supabaseServiceKey);
}
```

**Keep `createHash('sha1')`** — it's valid in Node.js backend. No need to move it to `server.ts`.

### Changes in `server.ts`

**Update imports:**

```typescript
// BEFORE:
import { getUserMemories, saveUserMemory, ... } from './src/services/userMemoryService';

// AFTER:
import { getUserMemories, saveUserMemory, ..., initUserMemoryService } from './server/services/userMemoryService';
```

**Initialize at startup:**

```typescript
// After env validation
initUserPreferencesService(supabaseUrl, process.env.SUPABASE_SERVICE_ROLE_KEY!);
initUserMemoryService(supabaseUrl, process.env.SUPABASE_SERVICE_ROLE_KEY!);
initUserTaskService(supabaseUrl, process.env.SUPABASE_SERVICE_ROLE_KEY!);
```

### `tsconfig.server.json` update

Add the new `server/` directory to the server compilation includes:

```json
{
  "include": ["server.ts", "server/**/*.ts"]
}
```

### Cleanup

- Delete `src/services/userPreferencesService.ts`
- Delete `src/services/userMemoryService.ts`
- Delete `src/services/userTaskService.ts`
- Keep `src/services/geminiService.ts` in place (it works in frontend — no Node deps)
- Remove empty `src/services/` directory if only `geminiService.ts` remains

## Phase 5: Feature Flag & Rollback

### `mima_security_v2` env var

| Value              | Behavior                                                                     |
| ------------------ | ---------------------------------------------------------------------------- |
| `"true"` (default) | Full security hardening: PG session store, AES-GCM, strict validation        |
| `"false"`          | Legacy mode: MemoryStore, AES-CBC with static salt, fallback secrets allowed |
| Not set            | Treated as `"true"` (hardened by default)                                    |

### Implementation pattern

```typescript
const SECURITY_V2 = process.env.MIMA_SECURITY_V2 !== 'false';

if (SECURITY_V2) {
  // Hardened path
  console.log('[SECURITY] Hardened mode enabled');
} else {
  // Legacy path
  console.warn('[SECURITY] Legacy mode — NOT hardened. Do not use in production.');
}
```

All 3 major changes (session store, encryption, startup validation) are gated behind this single flag.

## Risk Assessment

| Risk                                             | Mitigation                                                                   |
| ------------------------------------------------ | ---------------------------------------------------------------------------- |
| `connect-pg-simple` can't reach Supabase DB      | Validate `SUPABASE_DB_URL` at startup; pool connection test with 5s timeout  |
| Existing CBC-encrypted tokens can't be read      | `decrypt()` detects format by part count, falls back to CBC                  |
| Users lose sessions on deploy                    | Sessions are transient; Supabase JWT auth is independent of Express sessions |
| `pg` pool leaks connections                      | `pgPool.end()` on process exit; idle timeout 30s; max 10 connections         |
| Feature flag adds complexity                     | Single boolean check; minimal branching; clear logging of active mode        |
| TypeScript compilation fails after service moves | Update `tsconfig.server.json` includes; verify `tsc --noEmit`                |
